# Feature: Products

## Status

Approved — implementation refactored; migration awaiting manual developer generation

## Goal

Allow a warehouse operator to maintain a reliable catalogue of products that later inventory, purchasing, and sales workflows can reference.

## Scope

- Create a product with SKU, name, optional description, and active status.
- View a product by ID.
- Edit SKU, name, and description.
- View a paginated product list.
- Search the list by SKU or name.
- Activate and deactivate a product explicitly.
- Enforce SKU uniqueness in the application and PostgreSQL database.
- Provide server and client validation with useful error messages.

## Out of Scope

- Warehouses, balances, movements, stock adjustments, and reservations.
- Units of measure, pricing, cost, tax, categories, images, barcodes, and attachments.
- Product deletion, import/export, audit records, and authentication.
- Supplier-specific product data or purchase/sales document references.

## Assumptions

- A product consists only of the fields needed for the current milestone: SKU, name, optional description, and active status.
- SKU is trimmed and stored in uppercase. Uniqueness is case-insensitive, so `abc-001` and `ABC-001` are the same SKU.
- A new product is active by default. Deactivation prevents future operational use once later workflows exist; it does not delete historical data.
- Products are not deletable in this version. This avoids broken references when inventory and documents are introduced.
- Page numbering is one-based. The default page size is 20; clients may request 1 through 100 items.

## Business Rules

1. SKU is required, contains 1–64 non-whitespace characters after trimming, and is unique without regard to case.
2. Name is required, contains 1–200 non-whitespace characters after trimming.
3. Description is optional and limited to 1,000 characters when supplied.
4. SKU is stored in uppercase; name and description retain their user-entered casing after trimming.
5. A duplicate SKU is rejected with HTTP 409 Problem Details, including concurrent create requests.
6. Editing a missing product returns HTTP 404 Problem Details.
7. Activation and deactivation are idempotent: requesting the current status succeeds without changing other fields.
8. List search matches SKU or name case-insensitively. An omitted or blank search returns all products.

## Data Model Changes

Create a `Product` entity and EF Core configuration. After reviewing the model, the developer manually generates and applies the migration; Codex does not generate or edit migration files or the model snapshot.

| Field | Type | Rules |
|---|---|---|
| `Id` | `uuid` | Primary key; generated by the application. |
| `Sku` | `varchar(64)` | Required; uppercase; unique case-insensitively. |
| `Name` | `varchar(200)` | Required. |
| `Description` | `varchar(1000)` | Nullable. |
| `IsActive` | `boolean` | Required; defaults to `true`. |
| `CreatedAtUtc` | `timestamp with time zone` | Required UTC creation timestamp. |
| `UpdatedAtUtc` | `timestamp with time zone` | Required UTC modification timestamp. |

Database constraints and indexes:

- Primary key on `Id`.
- Unique index on `Sku`; the uppercase check constraint makes its uniqueness case-insensitive for the canonical stored SKU.
- Index on `Name` to support ordered list queries; use `UPPER("Sku")` and `UPPER("Name")` filtering for this small portfolio-scale catalogue.
- Check constraints preventing blank trimmed SKU and name values.

## API Requirements

All endpoints live under `/api/products`. Validation and conflict responses use RFC 7807 Problem Details.

| Method | Route | Purpose |
|---|---|---|
| GET | `/api/products?page=1&pageSize=20&search=...` | Return paginated, SKU/name-searchable products. |
| GET | `/api/products/{id}` | Return one product. |
| POST | `/api/products` | Create a product; return `201 Created`. |
| PUT | `/api/products/{id}` | Update SKU, name, and description. |
| PATCH | `/api/products/{id}/status` | Set active status explicitly. |

### Request contracts

`POST` and `PUT` request body:

```json
{
  "sku": "SKU-001",
  "name": "Sample product",
  "description": "Optional description"
}
```

Status request body:

```json
{
  "isActive": false
}
```

List response includes `items`, `page`, `pageSize`, and `totalCount`. Product responses include all persisted product fields.

- Add a Products navigation item to the application layout.
- Add `/products` list, `/products/new` create, `/products/:id` detail, and `/products/:id/edit` edit routes.
- Use Ant Design Table with server-side pagination and a SKU/name search field.

- Provide activate/deactivate actions with a confirmation dialog and refresh affected cached list/detail queries.

## Acceptance Criteria

### Scenario 1 — Create and find a product

Given no product uses SKU `SKU-001`
When an operator creates a product with SKU `sku-001` and a valid name
Then the API returns `201 Created`
And the stored and returned SKU is `SKU-001`
And the product appears in search results for both `sku-001` and its name.

### Scenario 2 — Reject a duplicate SKU

Given a product already uses SKU `SKU-001`
When an operator creates or edits another product with SKU `sku-001`
Then the API returns `409 Conflict` Problem Details
And the database contains only one product with that SKU.

### Scenario 3 — Validate form input

Given an operator is on the create or edit form
When they submit a blank SKU or name, or an overlong value
Then they see field-level validation errors
And no invalid request is persisted.

### Scenario 4 — Page and search the catalogue

Given more products exist than one page can contain
When an operator searches by a partial SKU or name and moves between pages
Then the table displays the matching server-side page and total count
And an empty search result has a clear empty state.

### Scenario 5 — Change status

Given an active product exists
When an operator deactivates it
Then its detail and list status display inactive
And activating it again restores its active status.

## Unit Tests

- Creating a product normalizes SKU and trims supported text fields.
- Blank SKU and name values are rejected.
- Overlong SKU, name, and description values are rejected.
- Product update preserves identity and updates `UpdatedAtUtc` in UTC.
- Status change is idempotent.

## Integration Tests

- `POST /api/products` creates and returns a product using PostgreSQL Testcontainers.
- Duplicate SKU is rejected with HTTP 409 by both application validation and the PostgreSQL unique index.
- `GET /api/products` paginates, searches SKU/name case-insensitively, and returns the total count.
- `GET` and `PUT` return 404 for an unknown ID.
- `PATCH /api/products/{id}/status` changes and returns the product status.
- EF Core migration creates the expected PostgreSQL constraints and indexes.

## Frontend Tests

- Product list renders loading, populated, empty, and API-error states using MSW.
- Product form blocks invalid input and renders server validation errors.
- Create mutation navigates or updates the list after a successful response.
- Deactivation confirmation sends the status mutation and updates visible status.
